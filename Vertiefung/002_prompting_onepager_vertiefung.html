A) Code-Generierung in R (Advanced Track)

Eigenständiger Track für Statistik-/Data-affine Anwender. Fokus auf ausführbaren Code, Datenvalidierung und reproduzierbare Skriptstruktur.

Use Case A – R-Code für deskriptiven Gruppenvergleich

Format: Markdown + Code-Spezifikation (Pakete, Datenstruktur, Regeln, Output)
Warum top: Pair-Programming mit KI statt „Code-Würfeln“

Die Vorlage zwingt zu sauberer Spezifikation: Pflichtspalten, Datentypen, erlaubte Pakete, Validierungslogik, Ausgabeformat und Risiken.

Struktur-Mehrwert: Der Code wird deutlich robuster, wenn Datenstruktur, Prüfregeln und gewünschte Ausgaben vorab strukturiert vorgegeben werden.
# Aufgabe
Erstelle R-Code für die unten beschriebene statistische Aufgabe.

## Ziel
- Korrekt ausführbarer R-Code
- Nachvollziehbar kommentiert
- Reproduzierbar (klare Pakete, klare Schritte)
- Robust gegenüber typischen Eingabefehlern (soweit sinnvoll)

## Kontext
- Zielgruppe: Statistikteam / Fachreferat
- R-Umgebung: R 4.3.x
- Erlaubte Pakete: dplyr, tidyr, ggplot2, readr
- Nicht verwenden: exotische Pakete / Internetzugriffe
- Datenquelle: CSV

## Eingabedaten / Datenstruktur
### Datenbeschreibung
- Datensatzname: ae_data
- Beobachtungseinheit: Fallmeldung
- Relevante Variablen:
  - patient_id (chr)
  - followup_type (factor: standard / extended)
  - completeness_score (numeric)
  - case_closure_days (numeric)

### Qualitätsregeln / Annahmen
- Fehlende Werte behandeln: in Berechnungen mit na.rm = TRUE
- Gruppierungslogik: followup_type

## Aufgabe im Detail
1. Daten validieren (Pflichtspalten prüfen)
2. Deskriptive Statistik nach followup_type berechnen
3. Ergebnis-Tabelle erstellen

## Ausgabeformat
### A) R-Code
- Vollständiges Skript in einem Block
### B) Kurze Erläuterung (max. 8 Bullet Points)

## Regeln
- Wenn Angaben fehlen, sinnvolle Annahme treffen und explizit markieren
- Kein Pseudocode, sondern ausführbarer R-Code

## Input
<<<ANFORDERUNG>>>
[Beschreibung der Analyseaufgabe]
<<<END>>>
ANFORDERUNG:
Wir möchten einen ersten Vergleich zwischen standard und extended follow-up durchführen.
Bitte erstelle ein R-Skript, das eine CSV einliest, Pflichtspalten prüft, Mittelwert und Median für completeness_score sowie Median case_closure_days berechnet und ausgibt.
# Pakete laden
library(dplyr)
library(readr)

# 1) Daten einlesen
ae_data <- read_csv("ae_followup.csv", show_col_types = FALSE)

# 2) Pflichtspalten prüfen
required_cols <- c("patient_id", "followup_type", "completeness_score", "case_closure_days")
missing_cols <- setdiff(required_cols, names(ae_data))
if (length(missing_cols) > 0) {
  stop(paste("Fehlende Pflichtspalten:", paste(missing_cols, collapse = ", ")))
}

# 3) Gruppenstatistik berechnen
summary_table <- ae_data %>%
  group_by(followup_type) %>%
  summarise(
    n_cases = n(),
    mean_completeness = mean(completeness_score, na.rm = TRUE),
    median_completeness = median(completeness_score, na.rm = TRUE),
    median_case_closure_days = median(case_closure_days, na.rm = TRUE),
    .groups = "drop"
  )
print(summary_table)

Praxis-Hinweis: Für Code-Prompts immer zuerst Datenstruktur + Pflichtspalten + erlaubte Pakete definieren. So wird die KI zum verlässlichen Pair-Programmer.

B) CustomGPT-Weiterentwicklung (Systemarchitektur)

Fokus: Systemprompt, Datenanbindung, Guidelines-Parsing – plus strukturierter User-Prompt. Ziel-Fähigkeit: „Regelbasierte Vorprüfung“.

Was gezeigt wird

  • Systemprompt (Instructions) als Basis
  • Einbindung von Knowledge-Dateien (Guidelines)
  • User-Prompts zur strukturierten Steuerung

Wichtige Botschaft

Ein gutes CustomGPT entsteht aus einer Prompt-Architektur: Instructions + Knowledge + strukturierter Fall-Input + Human Review.

Sicherheitsprinzip: Das GPT unterstützt die Vorprüfung – es trifft keine finale rechtliche Entscheidung.

Use Case B – Architektur für Regelabgleich & Berichte

Format: Markdown/XML-orientierte Prompt-Architektur
# Aufgabe

## Aufgabenpakete
1. Systemprompt-Architektur (Rolle, Regeln, Ausgaben)
2. Datenanbindung / Feldmapping
3. Guidelines-Parsing (Regelanwendung)
4. Qualitäts- und Freigabeprozess

## Regeln
- Trenne klar zwischen System-Logik, Daten-Logik und Textausgabe.

Use Case C – E-Commerce Sales Analyse (Q1 2026)

Format: Markdown-Prompt + Dateianhang (CSV) -> Code Execution
Warum top: End-to-End Automatisierung von Standard-Reports

Dem Prompt wird eine strukturierte Aufgabenstellung sowie eine (fiktive) Sales-CSV übergeben. Die KI schreibt das Python-Skript (pandas), führt es aus und liefert direkt die Geschäftsmetriken.

Struktur-Mehrwert: Klare Definition der Metriken (z. B. Formel für Conversion Rate) verhindert, dass die KI bei der Datenverarbeitung falsche Annahmen trifft.
# Aufgabe
Führe eine statistische Auswertung der beigefügten E-Commerce Sales-Daten durch. Schreibe den Python-Code und führe ihn (falls in der Umgebung möglich) direkt aus, um die Ergebnisse zu präsentieren.

## Ziel
- Deskriptive Auswertung der Sales-Performance für Q1 2026
- Nachvollziehbarer Python-Code (pandas)
- Strukturierte Ergebnisausgabe

## Eingabedaten
- Angehängte Datei: sales_q1_2026.csv
- Spalten: date, product_name, views, add_to_cart, purchases, revenue

## Aufgabe im Detail
1. Aggregiere die Daten auf monatlicher Basis (Jan, Feb, Mär).
2. Berechne folgende KPIs pro Monat:
   - Total Views
   - Total Revenue
   - Conversion Rate (Formel: Total Purchases / Total Views * 100)
   - Cart Abandonment Rate (Formel: (1 - (Total Purchases / Total Add_to_Cart)) * 100)
3. Identifiziere den umsatzstärksten Monat.

## Ausgabeformat
### A) Ergebnisse (Management Summary)
- Monatliche KPIs in einer Markdown-Tabelle
- Umsatzstärkster Monat hervorgehoben
### B) Verwendeter Python-Code
- Klar strukturierter und kommentierter Code

## Regeln
- Runden: Prozentwerte auf 2 Nachkommastellen, Währungen auf 2 Nachkommastellen.
- Keine Daten erfinden, nur die CSV nutzen.
DATEINAME: sales_q1_2026.csv
INHALT (Auszug):
date,product_name,views,add_to_cart,purchases,revenue
2026-01-15,SmartMug Pro,1500,300,150,5998.50
2026-01-28,SmartMug Pro,1200,250,110,4398.90
2026-02-14,SmartMug Pro,2500,600,300,11997.00
2026-02-22,SmartMug Pro,1800,450,200,7998.00
2026-03-05,SmartMug Pro,2100,500,250,9997.50
2026-03-20,SmartMug Pro,2800,700,350,13996.50
### A) Ergebnisse (Management Summary)

| Monat | Total Views | Total Revenue (€) | Conversion Rate (%) | Cart Abandonment Rate (%) |
|---|---|---|---|---|
| Jan 2026 | 2.700 | 10.397,40 | 9,63 % | 52,73 % |
| Feb 2026 | 4.300 | 19.995,00 | 11,63 % | 52,38 % |
| Mär 2026 | 4.900 | 23.994,00 | 12,24 % | 50,00 % |

**Fazit:** Der umsatzstärkste Monat in Q1 2026 war der **März** mit einem Gesamtumsatz von 23.994,00 €. Gleichzeitig verzeichnete der März die höchste Conversion Rate (12,24 %) und die geringste Cart Abandonment Rate (50,00 %).

### B) Verwendeter Python-Code
```python
import pandas as pd

# Daten laden
df = pd.read_csv('sales_q1_2026.csv', parse_dates=['date'])

# Monat extrahieren
df['month'] = df['date'].dt.to_period('M')

# Gruppieren und Aggregieren
monthly_stats = df.groupby('month').agg(
    total_views=('views', 'sum'),
    total_add_to_cart=('add_to_cart', 'sum'),
    total_purchases=('purchases', 'sum'),
    total_revenue=('revenue', 'sum')
).reset_index()

# KPIs berechnen
monthly_stats['conversion_rate'] = (monthly_stats['total_purchases'] / monthly_stats['total_views']) * 100
monthly_stats['cart_abandonment_rate'] = (1 - (monthly_stats['total_purchases'] / monthly_stats['total_add_to_cart'])) * 100

# Formatierung für Ausgabe vorbereiten
monthly_stats['month'] = monthly_stats['month'].astype(str)
```
